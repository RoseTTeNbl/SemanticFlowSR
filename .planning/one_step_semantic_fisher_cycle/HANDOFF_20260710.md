## Active algorithm flow

The current intended outer loop is a hard-endpoint population flow:

```text
theta0_i ~ mu0
  -> reference bridge atom sampler
  -> hard decoded expression endpoint population
  -> graph/expression exploration and mutation
  -> expression-level selection/tilt
  -> sharp active-graph projection back to theta1 atoms
  -> source-tracked semi-discrete Fisher/Hungarian coupling
  -> train v^{k+1}(theta_t, D, t, theta0) on analytic Fisher bridge tangents
```

Detailed semantics:

1. `theta0` randomness is the source of stochasticity.
2. Given a `theta0` and the current reference atom/coupling archive, endpoint generation is
   interpreted as deterministic hard expression decoding, not a soft endpoint distribution.
3. Iteration 1 uses compiled trace seeds. Iteration 2+ uses `previous_elite_archive` atoms.
4. `G` / one-step student is removed from the active path. `--cycle-one-step-student` raises.
5. `cycle_proposer_source=flow_rollout` is rejected; active source is `reference_bridge`.
6. Candidate generation is hard trace based:
   - hard argmax reference atom;
   - active-subgraph mutation;
   - term-readout mutation when `output_terms > 1`;
   - old soft endpoint categorical sampling is disabled unless explicit ablation flag is used.
7. Selection is expression-level, not graph-fiber-level:
   - graph-fiber duplicates are collapsed by decoded expression before projection;
   - selected rows record duplicate counts and representative graph choices.
8. Projection back to theta:
   - active blocks get epsilon-smoothed sharp one-hot targets;
   - inactive blocks preserve source identity;
   - Fisher/Hungarian coupling keeps tracked source particles, no fresh source replacement.

## Multi-term register readout

Current register graph supports `--output-terms 3` for medium runs.

Readout semantics:

- each readout slot selects an arbitrary register expression;
- final expression is the sum of unique nonzero readout terms;
- repeated/zero terms are collapsed at expression level for the summed expression, while
  diagnostics still record readout slots, duplicate terms, unique nonzero terms, and term
  expressions.

Compiler behavior:

- top-level `add` is flattened into up to `K=output_terms` term readouts when possible;
- examples covered by tests include `x0 + x0**2 + x0**3` and `sin(x0)*x0 + cos(x0)`;
- if term count exceeds `output_terms` or compilation fails, it falls back to single-root compile
  rather than silently truncating.

Eval metrics now separate:

- raw summed expression R2/NMSE;
- global affine fit R2/NMSE;
- term-level linear fit R2/NMSE;
- term counts, duplicate terms, per-term complexity;
- structural metrics computed on raw hard-decoded expression, not affine wrapper.

which hid the actual pushforward population.

## Loss interpretation

Do not treat `1e0` Fisher velocity loss as successful convergence.

The loss is blockwise Fisher tangent loss:

```text
sum_b w_b ||p_dot_pred,b - p_dot_target,b||^2_{p_t} / sum_b w_b
```

Observed values around `0.6-1.0` only mean better than zero prediction. They are still too large
for robust hard endpoint decoding. Endpoint quality must be judged by:

- terminal/readout max probability;
- decoded active probability;
- active argmax match;
- population multi-term rate;
- raw R2 and skeleton/operator-dependency;
- not fitted R2 alone.

Next useful diagnostics to add:

- relative Fisher loss vs zero-pred baseline;
- per-block-kind loss: readout/op/arg/inactive;
- endpoint sharpness by readout blocks specifically.

## Recommended next run

Do not start with a long medium. First add/inspect relative/per-block-kind losses, then run a
short overfit-style GPU job with strict gates.

Suggested validation gates:

- flow relative Fisher loss should drop materially below zero-pred baseline, not just absolute
  loss around `1e0`;
- readout max probability should increase over iterations;
- population multi-term rate should be nonzero on eval, but raw R2 must not collapse;
- fitted R2 cannot be used as success if raw/skeleton is bad;
- per-iteration eval should use `eval_task_limit >= 8` and `eval_theta0_samples >= 8` only after
  the shorter diagnostic run shows endpoint sharpness improving.

Suggested command template:

```bash
RUN_GPU=<free_gpu> SCALE=overfit \
TAG=<new_tag> \
CYCLE_ITERATIONS=2 TRAIN_LIMIT=8 EVAL_LIMIT=8 \
BOOTSTRAP_EPOCHS=8 BOOTSTRAP_STEPS=60 \
PARTICLES=8 EXPRESSION_SAMPLES=4 \
FLOW_EPOCHS=6 CYCLE_STEPS=60 \
HIDDEN=128 ODE_STEPS=32 ODE_SWEEP_STEPS= \
EVAL_THETA0_SAMPLES=8 EVAL_SAMPLES=1 \
READOUT_LOSS_WEIGHT=3.0 OP_LOSS_WEIGHT=1.5 \
bash scripts/run_one_step_semantic_fisher_cycle_gpu.sh
```

But do this only after adding relative/per-block-kind loss diagnostics; otherwise `1e0` loss
will continue to be ambiguous.
